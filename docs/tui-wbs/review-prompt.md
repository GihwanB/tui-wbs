# TUI WBS 프로젝트 다면 검토 프롬프트

이 프롬프트는 TUI WBS 프로젝트의 설계, 구현, 사용성을 다각도로 검토하기 위해 설계되었습니다. AI 모델이나 동료 검토자에게 이 프로젝트를 리뷰 요청할 때 사용하십시오.

---

## 🎭 역할 부여 (Role Definition)
당신은 **수석 소프트웨어 아키텍트(Senior Software Architect)**이자 **프로덕트 매니저(Product Manager)**, 그리고 **UX 전문가**의 관점을 모두 갖춘 종합 검토자입니다. 당신의 목표는 TUI WBS 프로젝트가 기술적으로 견고하고, 사용하기 편리하며, 원래의 기획 의도(PRD)를 충족하는지 비판적으로 분석하는 것입니다.

## 📋 검토 컨텍스트 (Context)
- **프로젝트명**: TUI WBS (Terminal User Interface Work Breakdown Structure Tool)
- **PRD 문서**: `docs/tui-wbs/tui-wbs.prd.md`
- **목표**: 터미널 환경에서 Markdown 기반의 WBS 파일을 트리 형태로 시각화하고 편집하는 도구 개발.
- **핵심 가치**:
    1. Markdown 파일의 Round-trip 무결성 보장 (읽기 -> 저장 시 원본 손상 없음)
    2. 개발자에게 친숙한 vim 스타일 키바인딩 제공
    3. 직관적인 TUI 제공
- **기술 스택**: Python 3.12+, Textual 8.x, Click (CLI 프레임워크)
- **프로젝트 단계**: `{프로젝트 단계 기입 — 예: Phase 1 MVP 설계 완료, 구현 전}`
- **검토 시점**: `{PRD 리뷰 / 코드 리뷰}` — PRD 리뷰 시 차원 4(코드 품질)는 건너뛰고, 코드 리뷰 시 전체 차원을 검토합니다.

## 🔍 검토 차원 (Review Dimensions)

다음 6가지 차원에서 프로젝트를 심층 분석해 주십시오.

### 1. 기능적 완전성 (Functional Completeness)
- **PRD 대조**: 구현된 기능이 PRD(Product Requirements Document)에 명시된 요구사항을 충족하는가?
- **누락된 기능**: WBS 관리 도구로서 필수적이지만 현재 설계에서 빠진 기능은 없는가? (예: 실행 취소/재실행, 필터링 등)
- **메타데이터 처리**: HTML 주석을 활용한 메타데이터 파싱/저장 로직이 다양한 에지 케이스(주석 내 특수문자 등)를 처리할 수 있는가?

### 2. 사용자 경험 (UX & Usability)
- **터미널 적합성**: TUI 레이아웃이 터미널 화면 크기 변화에 유연하게 대응하는가?
- **키바인딩 효율성**: vim 사용자에게 직관적인 키 매핑인가? 비-vim 사용자도 쉽게 배울 수 있는가?
- **시각적 피드백**: 상태 변경, 선택 이동, 에러 상황 등이 사용자에게 명확히 전달되는가?
- **접근성 (Accessibility)**: 색상 대비가 충분한가? 색각 이상 사용자를 위한 대안(아이콘, 텍스트 레이블)이 있는가? 스크린 리더 호환성을 고려했는가?

### 3. 기술적 아키텍처 (Technical Architecture)
- **구조적 건전성**: `Textual` 프레임워크를 효율적으로 사용하고 있는가? 컴포넌트 간 결합도는 적절한가?
- **확장성**: 향후 새로운 메타데이터 필드나 뷰(예: 간트 차트)가 추가될 때 구조를 얼마나 쉽게 확장할 수 있는가?
- **의존성 관리**: 외부 라이브러리 사용이 적절하며, 불필요하게 무거운 의존성은 없는가?
- **배포/패키징**: PyPI 배포 구조가 적절한가? CLI 엔트리포인트(`pyproject.toml`의 `[project.scripts]`)가 올바르게 설정되어 있는가? `pip install`로 설치 가능한가?

### 4. 코드 품질 및 유지보수성 (Code Quality)

> ⚡ **적용 조건**: 이 차원은 구현된 코드가 존재할 때에만 검토합니다. PRD 리뷰 단계에서는 건너뛰십시오.

- **가독성**: 코드가 명확하고 이해하기 쉬운가? 변수명과 함수명이 의도를 잘 드러내는가?
- **테스트 용이성**: 비즈니스 로직(파싱, 데이터 모델)이 UI 코드와 잘 분리되어 있어 단위 테스트가 용이한가?
- **에러 처리**: 잘못된 Markdown 포맷이나 파일 입출력 오류에 대해 적절히 방어하고 있는가?
- **테스트 전략**: 단위 테스트, 통합 테스트, E2E 테스트의 범위와 방법이 적절한가? 커버리지 목표(예: 핵심 파싱 로직 90%+)가 설정되어 있는가? 회귀 테스트를 위한 테스트 데이터(다양한 Markdown 에지 케이스)가 준비되어 있는가?

### 5. 신뢰성 및 안전성 (Reliability & Safety)
- **데이터 무결성 (Round-trip)**: 사용자가 수정하지 않은 주석이나 공백이 저장 시 유지되는가? (가장 중요한 요소)
- **대용량 처리**: 수천 라인의 Markdown 파일을 열었을 때 성능 저하가 없는가?
  - 성능 목표치 예시: 1,000라인 파일 로딩 < 2초, 5,000라인 파일 로딩 < 5초, 트리 노드 펼치기/접기 < 100ms
- **보안**: 파일 경로 주입(path traversal) 공격에 안전한가? 사용자가 지정하지 않은 임의 파일에 접근/덮어쓰기가 불가능한가? CLI 인자를 통한 명령어 주입이 방어되는가?

### 6. 외부 연구 및 벤치마킹 (External Research & Benchmarking)
- **유사 도구 검색 (GitHub/Web)**: `Taskwarrior`, `Kanban TUI`, `Markdown Editor` 등 유사한 TUI 도구들이 GitHub에 있는지 검색하고, 그들의 장점(기능, UX)을 벤치마킹할 수 있는지 제안해 주십시오.
- **라이브러리 추천**: 프로젝트 구현에 도움이 될만한 검증된 Python 라이브러리(PyPI)나 코드 스니펫이 있는지 웹 검색을 통해 찾아봐 주십시오. 특히 Markdown 파싱이나 TUI 위젯 관련 유용한 자료가 있는지 확인해 주십시오.
- **최신 트렌드 반영**: 최근 6개월 내의 TUI/CLI 트렌드(예: `Rich`, `Textual` 업데이트, 새로운 CLI UX 패턴)를 반영할 수 있는 요소가 있는지 확인해 주십시오.

## 📤 출력 형식 (Output Format)
검토 결과를 다음 형식으로 정리해 주십시오:

1.  **총평 (Executive Summary)**: 프로젝트의 현재 상태에 대한 간략한 요약 (강점/약점).
2.  **주요 발견 사항 (Key Findings)**:
    *   ✅ **강점 (Strengths)**
    *   ⚠️ **개선 필요 (Improvements Needed)** — 아래 우선순위 및 Phase 매핑 기준 적용
    *   🐞 **잠재적 버그/위험 (Potential Bugs/Risks)**

    **우선순위 분류 기준:**
    | 등급 | 의미 | Phase 매핑 |
    |------|------|-----------|
    | **P0** | 차단 이슈 — 이것 없이는 출시 불가 | Phase 1 (MVP) |
    | **P1** | 중요 — 초기 사용자 경험에 직접 영향 | Phase 1~2 |
    | **P2** | 개선 — 있으면 좋지만 나중에 해도 무방 | Phase 2~3 |

3.  **발견 사항 추적 테이블**: 각 발견 사항을 다음 형식으로 정리해 주십시오.

    | # | 차원 | 발견 사항 | 우선순위 | 상태 |
    |---|------|----------|---------|------|
    | 1 | 기능 | (예시) 실행 취소 기능 누락 | P0 | 미해결 |
    | 2 | UX | (예시) 에러 메시지 불명확 | P1 | 미해결 |

    상태: `미해결` / `해결` / `보류`

4.  **상세 권장 사항 (Detailed Recommendations)**: 각 검토 차원별 구체적인 개선 제안.
5.  **치명적 결함 점검 (Sanity Check)**: 데이터 유실 가능성이 있는 로직이 있는지 여부.
6.  **다음 단계 (Next Steps)**: 검토 결과를 바탕으로 즉시 수행해야 할 후속 조치 목록 (담당자, 기한 포함).
